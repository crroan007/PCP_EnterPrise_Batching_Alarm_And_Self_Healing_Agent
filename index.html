Section 02

Working Assumptions

These assumptions are based on research into the FilePro platform (fP Technologies, v6.1). Each will be validated during discovery. If wrong, the impact is noted.

# Assumption Impact if Wrong
AS1 FilePro runs as a visible Windows process detectable via tasklist or WMI If hosted inside a terminal emulator, detection strategy redesign needed
AS2 FilePro uses file-system-based I/O ‚Äî reads/writes to disk folders If purely database/socket-based, file monitoring is irrelevant
AS3 A failed process can be restarted by re-executing its launch command If restart requires manual lock-clearing or login, auto-recovery scope expands
AS4 Production machine has outbound internet access If air-gapped, need on-network relay or local-only alerting
AS5 FilePro does not require an interactive desktop session If active user session required, need auto-login + session-keep-alive wrappers
AS6 No existing monitoring solution in place If Nagios/Zabbix/PRTG exists, integrate rather than duplicate
AS7 ~2‚Äì3 hours with Brandy is sufficient for all discovery If IT team needed for server questions, timeline may slip
AS8 No FilePro source code modifications needed ‚Äî external wrapper only If scope creeps into modifying processing tables, fundamentally different project
AS9 FilePro is running version 6.x on Windows Older versions on Unix/Linux need different monitoring architecture
