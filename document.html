Configurability - Nothing Is Hardcoded

All mailboxes, folder names, output paths, and scheduling settings are fully configurable through the PCP Nexus Settings interface. Nothing is hardcoded into the application. The values shown throughout this document (mailbox addresses, folder names, network share paths, CSV placement time, auto-email recipients) are the current recommended defaults. PCP can change any of these settings at any time through the Settings interface without requiring a code update, a redeployment, or developer involvement.

Specifically, the following are all independently configurable per mailbox:

  • Mailbox address - which shared Outlook mailbox to monitor (e.g., eaffidavits@pcpusa.net, efiling@pcpusa.net)
  • Input folder name - where new emails are picked up from (e.g., AI To Be Completed, 1 - Accepted Filing)
  • Completed folder name - where processed emails are moved to (e.g., 3 - AI Completed)
  • Exception folder name - where errors and expired links are moved to (e.g., AI Exceptions)
  • PDF output folder path - where renamed or merged PDF files are saved (e.g., \\172.31.47.151\psaffidavits\AI_Inbound\File_Stamped_Affidavits)
  • CSV output folder path - where the daily CSV file is placed (e.g., \\172.31.47.151\psaffidavits\AI_Inbound)
  • CSV placement time - what time the daily CSV is generated and placed (e.g., 10:00 PM)
  • Auto-email recipient - who receives the daily CSV email (configurable, with on/off toggle)

1. System Overview

PCP Nexus is an automated document processing system that monitors two shared Microsoft Outlook mailboxes for accepted court filing notifications from Tyler Technologies. The system operates in two distinct phases:

Setting Phase 1 - eAffidavits (Single Document Processing) Phase 2 - eFiling Petitions (Multi-Document Merge Processing)
Mailbox eaffidavits@pcpusa.net efiling@pcpusa.net
Sender Email no-reply@efilingmail.tylertech.cloud no-reply@efilingmail.tylertech.cloud
Subject Line Contains Filing Accepted Filing Accepted
Input Document Prefixes (received in email) AX02, AX03, AX07, AX09 AX40, AXPB, AXPE, AXPM, AXPL, AXPA, AX69, AX81, CL
Output Document Prefixes (after rename) AX42, AX43, AX47, AX89 AX40 (merged petition document)
Processing Type Single document: download, rename, save Multiple documents: gather by Envelope ID, merge into one PDF, save
Important Clarification - Input vs. Output Prefixes:
Phase 1 receives documents with prefixes AX02, AX03, AX07, and AX09. These are the naming conventions in the emails as received from Tyler Technologies. After the bot downloads the file-stamped copy, it renames AX02 to AX42, AX03 to AX43, AX07 to AX47, and AX09 to AX89. The renamed prefixes (AX42, AX43, AX47, AX89) are the output naming conventions, not the input naming conventions.
Important Clarification - Phase 2 Ignored Documents:
Phase 2 receives documents with prefixes AX40, AXPB, AXPE, AXPM, AXPL, AXPA, AX69, AX81, and CL. Of these, AX69 (Misc Court Request), AX81 (Misc Court Request), and CL (Clerk Letter) are received but filed away only. These documents do not need to be saved, renamed, or merged with any other document. The bot will discard these prefixes during processing.

2.2 Phase 1 Workflow - Step by Step

Step 1. Bot opens Microsoft Outlook and navigates to the shared mailbox eaffidavits@pcpusa.net.
Step 2. Bot opens the input folder AI To Be Completed (single flat folder, no subfolders).
Step 3. Bot selects the oldest unread email in the folder.
Step 4. Bot validates: (a) Sender email address equals no-reply@efilingmail.tylertech.cloud, and (b) Subject line contains Filing Accepted. If either check fails, the email is moved to AI Exceptions.
Step 5. Bot reads the email body and identifies the Lead Document prefix. The prefix must be one of: AX02, AX03, AX07, or AX09. If the prefix does not match any of these four, the email is moved to AI Exceptions.
Step 6. Bot clicks the hyperlink in the email body to download the file-stamped PDF document from Tyler Technologies. If the download link has expired (links are active for 45 days) or is corrupted, the email is moved to AI Exceptions.
Step 7. Bot extracts the PCP Job Number from the Lead Document file name. The PCP Job Number is always 9 characters long, starts with the letter A, and follows the 4-character prefix. Example: from the file name AX02A25C05955.pdf, the PCP Job Number is A25C05955.
Step 8. Bot renames the downloaded file by replacing the input prefix with the corresponding output prefix, keeping the PCP Job Number unchanged:
Input Prefix (Received) Output Prefix (Renamed) Example Input File Name Example Output File Name
AX02 AX42 AX02A25C05955.pdf AX42A25C05955.pdf
AX03 AX43 AX03A25A05247.pdf AX43A25A05247.pdf
AX07 AX47 AX07A25A01870.pdf AX47A25A01870.pdf
AX09 AX89 AX09A25B02479.pdf AX89A25B02479.pdf
Note: Three of the output prefixes begin with AX4 (AX42, AX43, AX47). One output prefix begins with AX8 (AX89).
Step 9. Bot saves the renamed PDF file to the configured output folder (default: \\172.31.47.151\psaffidavits\AI_Inbound\File_Stamped_Affidavits).
Step 10. Bot marks the email as read and moves it to the completed folder 3 - AI Completed.
Step 11. Bot logs the processed email to the SQLite database and the PCP Nexus Dashboard.
Step 12. Bot saves the CSV spreadsheet to the configured CSV output folder (default: \\172.31.47.151\psaffidavits\AI_Inbound). See Section 4 for full CSV specifications.
Step 13. Bot checks for more unread emails in the input folder. If more emails exist, return to Step 3. If no more emails exist, the processing cycle is complete.

2.4 PCP Job Number Format

Rule Requirement Confirm
Total Length Always exactly 9 characters (including the leading letter A)
Starts With Always starts with the uppercase letter A
Characters Alphanumeric only (uppercase letters A-Z and digits 0-9). No dashes, underscores, extra spaces, or symbols.

3.2 Phase 2 Workflow - Step by Step

Step 1. Bot opens Microsoft Outlook and navigates to the shared mailbox efiling@pcpusa.net.
Step 2. Bot opens the input folder 1 - Accepted Filing.
Step 3. Bot scans the folder for unread emails. For each email, the bot validates: (a) Sender email address equals no-reply@efilingmail.tylertech.cloud, and (b) Subject line contains Filing Accepted. Emails that fail either check are moved to AI Exceptions.
Step 4. Bot reads the email body and extracts the Envelope ID (a unique identifier assigned by Tyler Technologies to group related documents). If no Envelope ID is found, the email is moved to AI Exceptions.
Step 5. Bot identifies the Lead Document prefix from the email body. The prefix determines how the document is handled:
  • Actionable prefixes (download and merge): AX40, AXPB, AXPE, AXPM, AXPL, AXPA
  • Ignored prefixes (discard - do not save or merge): AX69 (Misc Court Request), AX81 (Misc Court Request), CL (Clerk Letter)
If the prefix is AX69, AX81, or CL, the document is discarded and not included in the merge.
Step 6. Once the Envelope ID is identified, the bot searches the entire input folder for all other emails with the same Envelope ID. This ensures that all related documents for the same filing are gathered before merging, even if they arrived at different times.
Step 7. Bot waits for a 20-minute aggregation window to allow all related documents to arrive. After 20 minutes, the bot proceeds with whatever documents have been collected for that Envelope ID.
Step 8. Bot clicks the hyperlink in each email to download the file-stamped PDF document from Tyler Technologies. If any download link has expired (links are active for 45 days) or is corrupted, that email is moved to AI Exceptions.
Step 9. Bot sorts the downloaded documents for merging. The document with prefix AX40 (Petition) is always placed as Page 1 (the anchor document). All other actionable documents (AXPB, AXPE, AXPM, AXPL, AXPA) are merged after AX40 in the order received. Documents with ignored prefixes (AX69, AX81, CL) are discarded and not included.
Step 10. Bot merges all sorted documents into a single PDF file. The merged file is named using the format: the prefix AX40 (unchanged) followed by the PCP Job Number (9 characters, starting with the letter A), followed by .pdf. The PCP Job Number is extracted from the AX40 (Petition) document's original file name by removing the first 4 characters (the AX40 prefix).
Part Value Characters
Prefix AX40 4 characters
PCP Job Number A25C05955 9 characters (starts with A, alphanumeric)
Extension .pdf 4 characters
Full File Name AX40A25C05955.pdf 17 characters total
The AX40 prefix does not change (unlike Phase 1 where AX03 becomes AX43). The merged output keeps the AX40 prefix as-is.
Step 11. Bot saves the merged PDF file to the configured output folder (default: \\172.31.47.151\psaffidavits\AI_Inbound\File_Stamped_Petitions).
Step 12. Bot marks all related emails as read and moves all of them to the completed folder 3 - AI Completed. All emails for the same Envelope ID move together.
Step 13. Bot logs the processed filing to the SQLite database and the PCP Nexus Dashboard.
Step 14. Bot checks for more unprocessed Envelope IDs. If more exist, return to Step 4. If none remain, the processing cycle is complete.

6. Database Architecture - SQLite and MySQL

PCP Nexus uses a hybrid database strategy designed to minimize infrastructure requirements during initial deployment while providing a clear path to enterprise-grade database hosting when PCP is ready.

6.1 Initial Deployment - SQLite (Workstation Only)

Setting Value
Database Engine SQLite (file-based, no server required)
Location Stored locally on the workstation running PCP Nexus
Purpose Stores all processing logs, dashboard metrics, exception records, and CSV generation data
Accessibility Workstation-only. The database file lives on the same machine running the bot. Other workstations or users cannot query this database directly.
Advantage Zero additional infrastructure. No database server to install, configure, or maintain. PCP Nexus runs entirely self-contained on a single workstation.
Why SQLite First: SQLite allows PCP Nexus to be deployed and tested immediately without requiring PCP's IT team to provision a separate database server. This keeps the initial rollout simple and focused on validating the processing logic. Once the product is confirmed working in production, the transition to MySQL can be planned at PCP's convenience.

6.2 Long-Term Target - MySQL (Network Accessible)

PCP Nexus is architected as a hybrid SQLite/MySQL system. The application already supports both database engines. When PCP is ready to move to MySQL, the following will be needed:

# Requirement Description
1 MySQL Server Instance A MySQL server (version 5.7 or higher) installed on a server accessible from the workstation running PCP Nexus. This can be a dedicated server, a virtual machine, or even the same workstation if desired.
2 Database Credentials A MySQL username and password with permissions to create tables, insert records, and read data. PCP's IT team would provide these credentials.
3 Network Connectivity The workstation running PCP Nexus must be able to reach the MySQL server over the network (typically TCP port 3306).
4 Configuration Update Update the PCP Nexus Settings interface to enable MySQL as an additional database target. PCP Nexus supports running both SQLite and MySQL simultaneously - SQLite continues to serve as a local backup on the workstation while MySQL becomes the primary networked database. This is a settings change only - no code changes or redeployment required.

6.3 Data Migration - SQLite to MySQL

All data stored in SQLite can be easily ported to MySQL. The database schema (table structure, column names, data types) is identical between both engines. When PCP is ready to transition, all existing processing history, exception logs, and dashboard data accumulated during the SQLite phase will be migrated to MySQL automatically as part of the switchover. No data will be lost.

6.4 Benefits of MySQL (When Ready)

  • Multi-user access - Multiple workstations or users can query the processing database simultaneously
  • Centralized reporting - Dashboard data accessible from any machine on the network, not just the bot workstation
  • Backup and recovery - Enterprise-grade backup tools and replication options
  • Scalability - Handles larger data volumes as processing history grows over months and years
